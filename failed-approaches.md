# Failed Approaches — 共通失敗原則集 (sanitized)

本ファイルは過去のイテレーションで試した改善案の失敗から抽出された **汎用的な原則** のみを記載する。
具体的なベンチマークケース ID、特定 iter 番号、特定の失敗ケースに関する記述は含まれない。

新しい改善案を立てる際に必ず参照し、これらの原則に抵触する変更は提案しないこと。

## 共通失敗原則

1. **判定の非対称操作は必ず失敗する**: EQUIV か NOT_EQ の一方に有利な変更は、他方の正答率を犠牲にする。判定閾値の移動はゼロサムゲームであり、全体精度は改善しない。チェックリスト等のアドバイザリな形式での非対称な指示追加であっても、実質的な立証責任の非対称化を招く。

2. **出力側の制約は効果がない**: 「こう答えろ」「こう答えるな」という出力への制約（UNKNOWN 禁止、証拠閾値、比較対象除外）は、推論の質自体を改善しない。改善すべきは入力側（何を見るか）と処理側（どう考えるか）である。

3. **探索量の削減は常に有害**: このタスクでは情報不足が誤判定の主因であり、探索を減らす変更は全て悪化する。

4. **同じ方向の変更は表現を変えても同じ結果になる**: 表現やメカニズムが異なっても、効果の方向（例: NOT_EQ を出しにくくする）が同じなら結果も同じ。

5. **入力テンプレートの過剰規定は探索視野を狭める**: テンプレートで「何を記録するか」を厳密に規定すると、規定に合致する情報だけが前提に載り、規定外の重要な信号（副作用、例外、テストフロー等）が脱落する。前提収集の改善は「何を見るか」を限定するのではなく「どう考えるか」を改善する方向であるべき。

6. **「対称化」は既存制約との差分で評価せよ**: 既に片方向をカバーしている制約を「両方向に拡張」しても、既存方向には実効的変化がなく、新規方向にのみ制約が作用する。変更の効果は「変更後の文言」ではなく「変更前との差分」で決まる。対称的な文言であっても、差分が非対称なら効果も非対称になる。

7. **分析前の中間ラベル生成はアンカリングバイアスを導入する**: 分析ステップの前に変更の性質を記述・分類させると、AI は自己生成したラベルに後続の推論をアンカリングさせる。「自由記述」であっても、テンプレートのヒントや例示が暗黙のカテゴリを提供し、ラベルが判定方向へのショートカットとして機能する。中間ステップの追加が有効なのは、ラベル付けではなく、証拠の探索や検証を増やす方向の場合に限る。

8. **受動的な記録フィールドの追加は能動的な検証を誘発しない**: テーブルに「関係性を記述する列」を追加しても、AI はもっともらしいテキストを生成するだけで、実際の検証行動（コールパスのトレース、到達性の確認等）を追加的に行うわけではない。記録と検証は異なる認知操作であり、「何を書くか」を変えても「何を調べるか」は変わらない。さらに、記録オーバーヘッドが探索コストを増加させ、探索予算を圧迫する。推論の質を改善するには、記録テンプレートの拡張ではなく、検証行動を直接的に誘発する仕組み（例: 新しい探索ステップの追加）が必要。

9. **メタ認知的自己チェックは自己評価精度の限界により機能しない**: 「自分はトレースしたか？」等の自己チェックを分析プロセスに挿入しても、AI は自分の推論行為の質を正確に自己判定できない。さらに、自己チェックは「本当にやったか？」という疑念を注入するが、この疑念は積極的な結論（差異の提示を要する）に対してのみ遅延効果を持ち、消極的な結論（差異なし判断）には具体的な追加行動に繋がらない。結果として名目上対称なチェックが実効的には立証責任の非対称な引き上げとして作用する。自己チェックで推論の質を改善するには、AI の自己評価に依存せず、外部的に検証可能な行動を直接的に要求する必要がある。

10. **必要条件ゲートは失敗モードを弁別しない限り判別力を持たない**: 分析プロセスに条件分岐ゲート（YES/NO → 分岐）を挿入しても、ゲートの条件が判別したい失敗モードと直交していればゲートは形式的に通過されるだけである。検証ステップを挿入する際は「この条件が YES/NO で分かれるとき、判定結果は実際に変わるか？」を事前に検証すべき。さらに、通過率の高いゲートは記述オーバーヘッドのみを残し、探索予算を圧迫する（原則 #3 の間接的発動）。

11. **探索順序の固定は、証拠収集ではなく探索の偏りを生む**: 探索の開始順序をテンプレートで固定しても、必要な証拠が新たに増えるわけではない。むしろ最初に読む側に注意がアンカリングされ、その側の要約や記録が目的化しやすい。柔軟な探索が必要な場面で、片側起点を強制すると反対側の重要信号が後景化する。改善すべきは開始順序ではなく、往復の中で何を検証するかである。

12. **アドバイザリな非対称指示も実質的な立証責任の引き上げとして作用する**: チェックリストのような強制力のない形式であっても、片方向の結論を出す時にのみ高度な追加検証を要求すると、その結論を出すハードルが上がり、証明しきれない場合のフォールバックとして安全側への誤判定を誘発する。表現が「推奨」であっても、モデルはそれを自己証明の要件と受け取るため、結果として判定の非対称操作（原則 #1）を引き起こす。

13. **relevant test 集合の低精度な拡張は、探索増加でも有害である**: 「見落としを減らす」目的で探索対象を外側へ広げても、その拡張が relevant test の精度を落とすなら、広く拾った周辺要素が強い反例として誤採用される。問題は探索量の不足だけでなく、探索集合の境界が粗くなることにもある。有効な改善は単なる search scope 拡大ではなく、関連性の高精度な維持でなければならない。

14. **条件付きの特例探索を足しても、主比較ループを強化しなければ全体性能は下がる**: 特定症状でのみ発火する追加チェックは、一見すると探索の質向上に見える。しかしそれが中心ループ（relevant test の選定、差分から観測点までの因果追跡、counterexample の構成等）を直接改善しない場合、実際には低頻度のサイドクエストを増やすだけになる。改善案は「特例条件があるか」ではなく、「その手順が中心因果連鎖をどれだけ直接強化するか」で評価すべきである。

15. **意味論的な観測境界を、固定長の局所追跡ルールで近似してはならない**: 「N hop downstream を読め」のように、観測点までの追跡を固定ステップ数へ変換しても、その hop 数は call graph 上の便宜的な単位にすぎず、テストが実際に差を観測する境界と一致する保証がない。有効な改善は hop 数の指定ではなく、観測境界を意味論的に判別する形で中心因果連鎖を強化するものでなければならない。

16. **ネガティブプロンプトによる局所的な禁止は過剰適応を招く**: 「特定の状況から〜と推論するな」という禁止事項を追加すると、モデルはその文面を過剰に一般化し、正当な推論や証拠の採用まで手控えるようになる。また、禁止対象への注意が喚起されることで、本来注力すべき検証タスクから注意が削がれる。局所的な誤判定を防ぐために特定の推論パターンを禁止することは、実質的にその方向への立証責任を上げること（原則 #1）と同義であり、全体の精度を落とす。

17. **中間ノードの局所的な分析義務化はエンドツーエンドの追跡を阻害する**: 最終的な観測点ではなく、差分が発生した直後の中間状態の挙動に焦点を当てて分析を義務付けると、エージェントの注意がその中間ノードに固定される。本来必要なエンドツーエンドの因果追跡が弱まり、中間状態の挙動のみに基づく推論ジャンプを招く。

18. **特定の証拠カテゴリへの厳格な物理的裏付けの要求は、探索予算の枯渇を誘発する**: 誤判定を防ぐために特定の証拠にのみ `file:line` レベルでの明示的な裏付けを要求すると、その要件を満たすための追加の探索が各テストの分析ループに紛れ込み、本来の検証ループのコストを不必要に押し上げる。結果として、探索予算（ターン）が枯渇し、正当な推論まで結論に至らない UNKNOWN が激増する。

19. **エンドツーエンドの完全な立証義務の付与は、フレームワークの複雑さに直面して探索予算を枯渇させる**: すべての Claim について完全なコールパスの追跡と引用を要求したり、抽象的な推論原則において完全なエンドツーエンドの検証を強く義務付けたりすることは、複雑なフレームワークにおいて、中間層や抽象化を全てトレースするという持続不可能な探索負荷を生む。必要な検証ポイントを特定せずに一律のエンドツーエンドの証明を義務付けると、ターン枯渇により判定そのものが完了しなくなる。

20. **目標証拠の厳密な言い換えや対比句の追加は、実質的な立証責任の引き上げとして作用する**: 既存の表現を「より厳格・より排他的」な表現に書き換えることは、意図が明確化であっても、モデルに強い警告として受け取られる。結果として、正当な証拠に基づく推論まで躊躇させ、正解を別の判定や UNKNOWN へと落とす。これは原則 #1（判定の非対称操作）と #16（ネガティブプロンプトの過剰適応）の複合である。

21. **無目的化された探索空間の拡大は、制約の存在以上に有害である**: 有害と判断された制約を単に削除（元の状態に差し戻し）して探索空間を広げるだけでは、過去の高スコア状態には戻らない。モデルは制約と共に「何を優先して探索するか」という文脈や焦点を失い、無秩序な探索に陥ってターン予算を枯渇させる。制約を解除する際は、単なる削除ではなく、探索の方向性を導くための適切なナビゲーションや代替の足場を用意しなければならない。

22. **抽象原則での具体物の例示は、論理的指標ではなく物理的探索目標として過剰適応される**: 抽象的な推論原則の中で、推論の終点や考慮事項として特定のコード要素を明示すると、エージェントはそれを「必ず発見して読解・引用すべき物理的なターゲット」と解釈する。その結果、その対象の特定が難しい環境において、本質的な因果推論の代わりに不毛な検索にターンを浪費し、探索予算の枯渇を招く。具体物ではなく、状態や性質（例: final outcome, observable effect）で指示すべきである。

23. **具体的な検証手順を伴わない抽象的な問い（ソフトフレーミング）は、推論を改善せず認知負荷のみを増やす**: 「常に X の影響を問え」というようなソフトなフレーミングをテンプレート実行前に置いても、具体的な検証方策がセットでなければ、エージェントの推論プロセス自体は変化しない。結果として、抽象的な目標への注意喚起だけが行われ、過剰適応や認知負荷によるターン消費を引き起こす。

24. **収束条件（探索停止）への物理的証拠の必須化は、探索予算を枯渇させる**: 探索を停止するためのゲートにおいて具体的な物理的証拠（`file:line` の引用など）を義務付けると、継続と停止のハードルが対称化され推論の質が上がると期待される。しかし実際には、結論の方向性が見えている状態から「引用フォーマットを満たすための証拠探し」という付加的な探索行動を強いることになり、結果としてターン予算を圧迫・枯渇させる（原則 #18 の収束判定への適用）。

25. **事前検証手順（Pre-check）の義務化は、本丸の推論パスを遠ざける**: テスト候補を選定した直後に、その候補の関連性（例: コールチェーンでの到達確認）を個別に事前検証（confirm）させる手順を追加すると、一見すると「どう探すか」の精度が上がり、偽陽性/偽陰性が減るように思われる。しかし、エージェントは各候補に対して個別に深いトレース探索を開始してしまい、有限のターン予算を事前検証だけで使い果たす。関連性の精査は、候補選定と主たる分析ループ（差分の影響追跡）の中で統合的に行われるべきであり、独立したゲートとして切り出すべきではない。

26. **中間ステップでの過剰な物理的検証要求は、探索予算の枯渇と安全側への誤判定を誘発する**: エッジケース等の中間分析ステップで「結論の根拠となる特定のコード要素（例: テストアサーション）の命名」を対称的に義務付けると、因果連鎖が完結し精度が上がるように思われる。しかし実際には、エージェントに対して「コードを再検索して対象を特定する」という能動的な検証行動を各ケースで強制することになり、探索予算の枯渇を招く。また、対象を特定しきれない場合には安全側（EQUIV等）への誤判定のフォールバックを引き起こす。これは原則 #18（特定証拠への物理的裏付け要求）の変種であり、中間ステップでの具体的証拠の要求は、全体の因果推論をかえって阻害する。
