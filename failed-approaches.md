# Failed Approaches（試行済み・失敗した改善方向）

本ファイルは過去のイテレーションで試行し、スコアを悪化させた改善方向を記録する。
新たな改善案を立てる際に必ず参照し、同じ方向の変更を繰り返さないこと。

## ブラックリスト

### BL-1: テスト削除を ABSENT として定義追加
- 試行: iter-6, 10, 15
- 内容: 削除されたテストは FAIL ではなく ABSENT として扱う定義（D3等）を追加
- 結果: NOT_EQ の正答率が低下（90% → 70%）
- 原因: ABSENT 定義が NOT_EQ 判定のハードルを上げすぎ、本来 NOT_EQ であるケースを YES に倒した
- Fail Core: この変更の本質は「比較対象から特定のテストを除外するルールの追加」である。テストを比較対象から除外すること自体が NOT_EQ の証拠を減らす方向に作用する。つまり「何を比較しないか」を定義する変更は、比較の網羅性を下げ、差異の検出力を弱める。EQUIV の誤判定を防ごうとして「これは比較しなくてよい」というルールを導入すると、本来比較すべきケースまで除外してしまう。

### BL-2: NOT_EQ 判定の証拠閾値・厳格化
- 試行: iter-7, 8, 13, 16, 17, 19, 20
- 内容: counterexample にアサーションまでのトレースを要求、仮想環境での反例を禁止、call path の明示を要求、等
- 結果: スコア 65-75% で安定せず
- 原因: NOT_EQ と判定すべきケースで証拠不十分として YES に倒れる。真の NOT_EQ の検出力を弱めた
- Fail Core: この変更の本質は「NOT_EQ と結論するための立証責任を引き上げる」ことである。立証責任が高いほど、エージェントは証拠が十分でないと判断して YES（EQUIV）に倒れやすくなる。NOT_EQ の閾値を上げることと EQUIV の閾値を下げることは表裏一体であり、一方を厳しくすれば他方に判定が流れる。モデルの推論能力が一定である以上、閾値の操作はトレードオフを移動させるだけで、全体の精度を上げることにはならない。

### BL-3: UNKNOWN 回答の禁止
- 試行: iter-8, 9, 11, 21
- 内容: UNKNOWN は許さず LOW confidence でも YES/NO を出すよう強制
- 結果: 改善効果なし
- 原因: 強制的に回答させても判定精度は上がらない。むしろ根拠の薄い判定が増える
- Fail Core: この変更の本質は「判断を保留する選択肢を奪い、不確実な状況でも結論を強制する」ことである。UNKNOWN が出るのはエージェントが十分な証拠を収集できなかった結果であり、そこで結論を強制しても判断材料が増えるわけではない。問題は「結論を出せないこと」ではなく「十分な証拠を集められないこと」にある。結論の強制は症状への対処であり、原因（証拠収集プロセスの質）への対処ではない。

### BL-4: 早期打ち切り（CONVERGENCE GATE）
- 試行: iter-12, 14
- 内容: 確信が得られた時点で探索を打ち切る STOP / CONVERGENCE GATE の追加
- 結果: 改善効果なし
- 原因: 探索を短縮すると見落としが増え、精度に寄与しない
- Fail Core: この変更の本質は「探索量を減らして効率を上げようとする」ことである。しかしこのタスクの誤判定は探索不足（見落とし）から生じるケースが多く、探索を早く打ち切ることは問題を悪化させる。「十分に見た上で間違える」のではなく「十分に見ないから間違える」場面では、探索の効率化ではなく探索の質の向上が必要。確信度で打ち切ると、偽の確信（誤った思い込み）を確定させてしまう。

### BL-5: P3/P4 をテストソースのアサーション条件記録形式に強化
- 試行: iter-6
- 内容: Compare テンプレートの PREMISES P3/P4 を「テストソースを読み、各テストのアサーション条件を [test name]: asserts [exact condition] at [file:line] の形式で記録せよ」に変更。分析前の前提品質を向上させ、EQUIV 誤判定（コード差異→テスト結果差異の推論ジャンプ）を防ぐ狙い
- 結果: スコア 85% → 75%（EQUIV 改善なし、NOT_EQ が回帰）
- 原因: P3/P4 に具体的なアサーション行の記録を要求したことで、AI の注意が個別のアサーション行に固定された。NOT_EQ ケースではテストの失敗がアサーション行だけでなく、セットアップ・副作用・例外・コードフロー全体から生じるが、テンプレートが「asserts [exact condition] at [file:line]」を要求するため、これらの信号が前提から漏れた。結果、本来 NOT_EQ と判定すべきケースで証拠不足として EQUIV に倒れた
- Fail Core: この変更の本質は「前提収集テンプレートに具体的な形式と対象を規定し、AI が見るべき情報を限定する」ことである。テンプレートが「何を記録するか」を厳密に指定するほど、指定外の情報（副作用、例外伝播、テストフロー等）は前提として記録されにくくなる。前提が狭まれば、そこから導出される分析も狭まり、テンプレートの想定外の経路で生じる差異を見落とす。これは既存原則「探索量の削減は常に有害」（BL-4）の変種であるが、探索の「量」ではなく「視野」を制約する点が異なる。新たな原則として記録する。

### BL-6: Guardrail 4 の「対称化」（両方向に trace 義務を拡張）
- 試行: iter-7
- 内容: Guardrail 4 の「差異がないと結論する前に trace せよ」を「差異があると結論する前にも trace せよ」に拡張し、EQUIV・NOT_EQUIV 両方向に trace 義務を課すよう対称化した。EQUIV 偽陽性 3 件（15368, 13821, 15382）がコード差異発見後にテスト結果への影響検証なく NOT_EQUIV と結論するジャンプを防ぐ狙い
- 結果: スコア 75% → 70%
- 原因: 「対称化」と銘打ったが、元の Guardrail 4 は既に EQUIV 方向（差異がないと結論する前に trace）をカバーしていた。したがって実質的な新規制約は NOT_EQUIV 方向にのみ追加された。NOT_EQUIV と結論する前に trace を義務化することで、真の NOT_EQUIV ケースで証拠不十分と判断して UNKNOWN/EQUIV に流れるケースが増加し、EQUIV 偽陽性の改善なく NOT_EQUIV の正答率が低下した
- Fail Core: この変更の本質は「既に片方向をカバーしている制約を両方向に拡張する」ことだが、既にカバー済みの方向には実効的変化がなく、新規に追加された方向にのみ制約が作用する。結果として BL-2 と同じ「NOT_EQ の立証責任引き上げ」が発生する。「対称化」「両方向均等」という表現で覆われていても、既存制約との差分で見れば片側への制約追加に過ぎない。これは共通原則 #4「同じ方向の変更は表現を変えても同じ結果になる」の典型例であり、さらに新たな原則を示唆する: **既存制約との差分が一方向にしか作用しない変更は、表現上「対称」であっても実効的には非対称である**。

### BL-7: CHANGE CHARACTERIZATION（分析前の変更性質の自由記述ステップ追加）
- 試行: iter-6
- 内容: Compare テンプレートの PREMISES と ANALYSIS の間に「各変更が何を変えているかを自由記述で確認するフェーズ」を4行追加。`Change A: [what it changes and where — production code / test code / both]` の形式で、判定方向を誘導するルールを含まない「純粋な認知的足場」として設計
- 結果: スコア 85% → 75%
- 原因: 「自由記述」と銘打っているが、テンプレートが `production code / test code / both` というヒントを提示することで、実質的に三択の暗黙カテゴリが導入された。AI は分析前に「Change B はテストコードのみを変更する」等の中間ラベルを自己生成し、そのラベルが後続の ANALYSIS をアンカリングした。「テストコードのみ」→ EQUIV 方向、「プロダクションコード」→ NOT_EQ 方向という暗黙のヒューリスティックが形成され、テスト動作の実際のトレースに基づく判定がラベルによるショートカットに置き換わった。結果としてベースラインで正答していたケースが回帰した
- Fail Core: この変更の本質は「分析前に変更の性質を中間ラベルとして自己生成させる」ことである。自由記述であっても、テンプレートのヒント（production code / test code / both）が暗黙のカテゴリを提供し、AI はそのカテゴリに沿ったラベルを生成する。生成された中間ラベルは後続の推論に対するアンカーとして機能し、証拠に基づく分析をラベルに基づくショートカットに変質させる。これは原則 #1（判定の非対称操作）の変種であるが、ルールによる直接的な誘導ではなく、AI 自身が生成した中間ラベルを経由する間接的な誘導である点が異なる。新たな原則として記録する。

### BL-8: Step 4 トレーステーブルに `Relevant to` 列を追加（localize の RELEVANT 列の移植）
- 試行: iter-7
- 内容: Step 4: Interprocedural tracing のテーブルに `Relevant to` 列（「この関数の振る舞いがどのテスト・前提・仮説に関係するか」を記録）を追加。localize モードの Phase 2 テーブルが持つ `RELEVANT` 列を共通コアに移植し、探索中にコード差異とテストの接続を明示させることで、テスト到達性を確認せずに NOT_EQUIVALENT と結論する「浅い反例」パターンを防ぐ狙い
- 結果: スコア 75% → 70%（EQUIV 偽陽性 3 件は改善なし、NOT_EQUIV 側で django__django-11433 が正答→UNKNOWN に回帰）
- 原因: EQUIV 偽陽性（15368, 13821, 15382）について、`Relevant to` 列は改善効果がなかった。エージェントはコード差異を発見した関数に対し「Relevant to: test_xxx call path」等のもっともらしい記述を生成するが、それは到達性の**検証**ではなく**記録**に過ぎず、推論プロセスを変えなかった。一方、NOT_EQUIV 側では django__django-11433 が iter-6 で正答（23 turns）→ iter-7 で UNKNOWN（31 turns）に回帰した。各関数に対する追加列の記入オーバーヘッドがターン消費を増加させ、探索予算を圧迫した結果、結論に到達できなかった
- Fail Core: この変更の本質は「探索中に関係性を記録する列を追加し、それにより推論の質が向上する」という仮説に基づくが、**受動的な記録フィールドの追加は能動的な検証を誘発しない**。エージェントは新しい列にもっともらしいテキストを生成するだけで、到達性の実際の検証（コールパスのトレース等）を追加的に行うわけではない。記録と検証は異なる認知操作であり、「何を書くか」のテンプレートを追加しても「何を調べるか」は変わらない。さらに、記録オーバーヘッドが1関数あたりのコストを増加させ、探索可能な関数数を実質的に減らす（原則 #3 の間接的な発動）。結果として、狙った EQUIV 改善は得られず、NOT_EQUIV 側でターン枯渇による回帰が発生した。これは既存原則の組み合わせ（記録 ≠ 検証 + 探索量の間接的削減）であるが、「記録フィールドが推論を改善する」という仮説の否定として新たな原則を記録する。

### BL-9: Trace check 自己チェック行の追加（メタ認知的自己評価）
- 試行: iter-6
- 内容: Compare テンプレートの ANALYSIS OF TEST BEHAVIOR 内、各テストの `Comparison:` 行の直後に `Trace check: [TRACED / INFERRED]` の自己チェック行を追加。INFERRED と自己評価した場合に「trace before proceeding」という行動指示で追加トレースを促す。fail-to-pass / pass-to-pass 両ブロックに同一の 3 行を追加（計 6 行追加、既存変更なし）
- 結果: スコア 85% → 75%（持続的 EQUIV 偽陽性 3 件 [15368, 13821, 15382] は改善なし。NOT_EQ 側で 14787 が正答→EQUIV に回帰、12663 が正答→UNKNOWN [31 turns] に回帰）
- 原因: 持続的 EQUIV 偽陽性 3 件について、AI はコード差異を発見した時点で「差異のあるコードパスをトレースした」と自己評価し TRACED と記入した。AI にとって「コード構造を読んで差異を確認する行為」と「具体的なテスト入力をコードパスに通してトレースする行為」の区別は曖昧であり、自己チェックは常に TRACED に倒れた。結果、INFERRED → 追加トレースという意図したメカニズムが発動しなかった。一方、NOT_EQ 側では Trace check が追加の自己疑念を注入した。14787 では差異を発見した後に Trace check で再検証を促され、過剰な慎重さから EQUIV に倒れた。12663 では Trace check による追加トレースの試行がターン消費を増加させ、31 turns で結論に到達できず UNKNOWN に終わった
- Fail Core: この変更の本質は「自分の推論行為を自己評価させるメタ認知的チェックポイントの挿入」である。しかしメタ認知的自己評価には2つの構造的問題がある。(1) **自己評価の精度問題**: AI は自分がトレースしたかどうかを正確に自己判定できない。コード差異の確認（コードを読む行為）と具体的入力のトレース（テストケースをコードパスに通す行為）は AI の内部では区別が曖昧であり、自己チェックは常に「やった」（TRACED）に倒れる。(2) **疑念注入の非対称性**: 自己チェックは「本当にやったか？」という疑念を注入するが、この疑念は結論を遅延させる方向にのみ作用する。NOT_EQ の結論は具体的な差異の提示を要求するため疑念の影響を受けやすく、EQUIV の結論は「差異がない」という消極的判断であるため疑念が具体的な追加行動に繋がりにくい。結果として BL-2/BL-6 と同じ「NOT_EQ の実質的な立証責任引き上げ」が発生する。既存原則 #1（判定の非対称操作）、#6（対称化の実効差分）に該当する。さらに新たな原則を示唆する: **メタ認知的自己チェックは自己評価精度の限界により意図した効果を発揮せず、疑念注入による結論遅延として作用する**。

### BL-10: Reachability ゲート（テスト到達性の条件分岐ステップ追加）
- 試行: iter-7
- 内容: Compare テンプレートの ANALYSIS OF TEST BEHAVIOR 内、各テスト分析ブロックの先頭に「このテストのコールパスは変更コードに到達するか？」を YES/NO で判定する Reachability ステップを挿入。NO なら即 SAME でスキップする条件分岐ゲートとして設計。fail-to-pass / pass-to-pass 両ブロックに同一の 4 行を追加（計 8 行）
- 結果: スコア 75% → 70%（EQUIV 偽陽性 3 件 [15368, 13821, 15382] は改善なし。NOT_EQUIV 側で 11433 が正答 [23 turns] → UNKNOWN [31 turns] に回帰、14787 は EQUIV [21 turns] → UNKNOWN [31 turns] に変化し依然誤答）
- 原因: EQUIV 偽陽性 3 件について、Reachability ゲートは効果がなかった。これらのケースではテストのコールパスは実際に変更コードに到達しており（relevant test として選ばれている以上、到達性は大抵 YES である）、エージェントは「YES — [file:line]」と記入してゲートを通過し、従来と同じ推論で NOT_EQUIVALENT と結論した。Reachability（到達するか）は EQUIV/NOT_EQUIV の判別に必要な条件だが十分条件ではない——到達していてもテスト結果が変わらないケース（EQUIV 偽陽性の本質）を識別できない。一方、NOT_EQUIV 側では各テストに対する Reachability 欄の記入（file:line 引用含む）がターンオーバーヘッドとなり、11433（元 23 turns）が 31 turns でターン上限に達して UNKNOWN に回帰した
- Fail Core: この変更の本質は「分析プロセスに条件分岐ゲートを挿入し、能動的な検証を促す」ことだが、ゲートの条件（テストが変更コードに到達するか）が判別したい失敗モード（到達しているが結果が変わらないケース）と直交している。relevant test は定義上変更コードのコールパス上にある（D2 で「コールパスにある場合のみ relevant」と規定）ため、Reachability は relevant test に対してほぼ常に YES となり、判別力を持たない。BL-8 の教訓（「受動的記録ではなく能動的検証を」）に従い能動的ゲートを設計したが、ゲート条件自体が失敗モードを弁別しなければ、ゲートは形式的に通過されるだけで推論プロセスを変えない。さらにゲート通過のための記述オーバーヘッドが探索予算を圧迫する（原則 #3 の間接的発動）。これは既存原則の組み合わせでは完全に捕捉できない新たなパターンを示す。

### BL-11: `ANALYSIS OF TEST BEHAVIOR` に outcome mechanism 注釈を追加
- 試行: iter-23
- 内容: Compare テンプレートの `ANALYSIS OF TEST BEHAVIOR` 冒頭に、「pass/fail を決める具体的メカニズム（assertion / raised exception / setup/teardown failure / side effect）をトレースせよ。単一 assert に還元するな」という注釈を 1 文追加した
- 結果: スコア 85% → 70%
- 原因: 意図は「assert 行だけを見るな」という視野拡張だったが、実際には `concrete mechanism` と列挙された観点が新たな注視先として働き、各テストで outcome mechanism を明示的に同定する作業負荷を増やした。その結果、エージェントは変更差分からテスト結果への因果連鎖を素直に追う代わりに、テスト側の失敗メカニズムのラベル付け・説明に注意を吸われ、既存の正解ケースでも過剰にテスト内部へ寄る/根拠を言い過ぎる方向にぶれた。つまり「視野を広げる」つもりの注釈が、実際には新しい記述フレームへのアンカリングとして働き、分析の自由度を下げた
- Fail Core: この失敗の本質は「入力テンプレート上で注目すべき観点を具体的に言い換えると、その観点自体がアンカーになり、モデルの探索をその枠に閉じ込める」ことである。assert 固定を避けるために outcome mechanism を列挙しても、列挙された観点に沿った説明を作ることが目的化し、変更コード→実行経路→テスト結果という本来の因果追跡が弱まる。これは原則 #5『入力テンプレートの過剰規定は探索視野を狭める』の別形態であり、新たな原則追加は不要である。

### BL-12: テストソース先読みによる固定順序化（`Entry:` フィールド追加）
- 試行: iter-24
- 内容: Compare テンプレートの `ANALYSIS OF TEST BEHAVIOR` に「各 relevant test では、変更コードを追う前にテストソースを読んで entry point・arguments・setup を確定せよ」という順序制約を追加し、各テスト分析ブロックに `Entry:` フィールドを新設した。さらに `Claim C[N].1` / `Claim C[N].2` を `starting from Entry` / `trace from Entry through changed code` に変更し、各トレースを Entry 起点で書かせる形にした
- 結果: スコア 70% → 65%
- 原因: 変更差分→テスト結果の因果を直接追わせる代わりに、「まずテストの入口を確定する」という固定手順が新たな作業目標になった。これによりエージェントの注意は test entry / arguments / setup の要約に引かれ、肝心の「変更された意味論がそのテスト入力で結果差に繋がるか」の比較が後景化した。また、順序を固定しても誤判定の本体である推論ジャンプ自体は消えず、単に test-first の枠組みの中で再発した。加えて、各 relevant test ごとに `Entry:` の抽出と file:line 記録が必要になり、探索コストも増えたため、改善が得られないまま全体スコアが下がった
- Fail Core: この失敗の本質は「正しい推論に必要な証拠を増やす代わりに、探索の開始順序を固定し、その順序に沿った中間記述を必須化する」ことである。探索順序の固定は、双方向に往復しながら差分とテストを突き合わせるべき問題を一方向の手順へ押し込み、最初に読んだ側（ここではテスト入口）へのアンカリングを生む。さらに `Entry:` の記述は外形上は事実確認だが、実際には「まずこれを埋める」が目的化し、比較推論そのものを強化しない。既存原則では #5『入力テンプレートの過剰規定』と #8『受動的記録は検証を誘発しない』に近いが、今回の失敗は「何を書くか」だけでなく「どちら側から読むか」を固定した点が本質であり、新たな原則追加を検討すべきである

### BL-13: `Key value` データフロートレース欄の追加
- 試行: iter-25
- 内容: Compare テンプレートの `ANALYSIS OF TEST BEHAVIOR` で、各 relevant test の `Claim C[N].1 / C[N].2` の前に `Key value` フィールドを追加し、assertion を決める 1〜2 個の変数・返り値について A/B それぞれ `created → last modified → value at assertion` を file:line 付きで記録させた
- 結果: スコア 85% → 75%
- 原因: 意図は「assertion 到達時の具体値」を両変更で対称的に追わせることだったが、実際には「どの 1〜2 個を key value とみなすか」を先に決めて埋める作業が新たなアンカーになった。テスト結果はしばしば単一変数ではなく、複数条件・制御フロー・例外・副作用の合成で決まるが、テンプレートが key value の選定と `created/modified/value` の記述を要求するため、エージェントはその枠に収まる代表値をもっともらしく記録することに注意を使い、枠外の因果経路を追いにくくなった。結果として、比較推論そのものが強化される代わりに、狭い代理表現への要約と記録が目的化し、正答ケースまで取りこぼした
- Fail Core: この失敗の本質は「比較対象の意味論を、テンプレートが指定する少数の代理変数へ圧縮させる」ことである。表面上は A/B 両側に同じ義務を課していても、実際に追加されたのは『何を key として表現するか』という入力表現の固定であり、その固定が探索視野を狭める。さらに `created → modified → value` 欄は検証行動そのものではなく記録欄なので、AI は適切な key value を本当に発見したかを保証されないまま、もっともらしいトレースを書けてしまう。したがって本件は新原則を要するというより、既存の原則 #5『入力テンプレートの過剰規定は探索視野を狭める』と #8『受動的な記録フィールドの追加は能動的な検証を誘発しない』の複合再発である

## 共通の失敗パターン

上記の Fail Core から導出される共通原則:

1. **判定の非対称操作は必ず失敗する**: EQUIV か NOT_EQ の一方に有利な変更は、他方の正答率を犠牲にする。判定閾値の移動はゼロサムゲームであり、全体精度は改善しない。
2. **出力側の制約は効果がない**: 「こう答えろ」「こう答えるな」という出力への制約（UNKNOWN 禁止、証拠閾値、比較対象除外）は、推論の質自体を改善しない。改善すべきは入力側（何を見るか）と処理側（どう考えるか）である。
3. **探索量の削減は常に有害**: このタスクでは情報不足が誤判定の主因であり、探索を減らす変更は全て悪化する。
4. **同じ方向の変更は表現を変えても同じ結果になる**: BL-1 の「ABSENT 定義」も BL-2 の「証拠閾値」も、本質は「NOT_EQ を出しにくくする」という同一の効果を持つ。表現やメカニズムが異なっても、効果の方向が同じなら結果も同じ。
5. **入力テンプレートの過剰規定は探索視野を狭める**: テンプレートで「何を記録するか」を厳密に規定すると、規定に合致する情報だけが前提に載り、規定外の重要な信号（副作用、例外、テストフロー等）が脱落する。探索の量ではなく視野の制約であり、BL-4 とは独立に作用する。前提収集の改善は「何を見るか」を限定するのではなく「どう考えるか」を改善する方向であるべき。
6. **「対称化」は既存制約との差分で評価せよ**: 既に片方向をカバーしている制約を「両方向に拡張」しても、既存方向には実効的変化がなく、新規方向にのみ制約が作用する。変更の効果は「変更後の文言」ではなく「変更前との差分」で決まる。対称的な文言であっても、差分が非対称なら効果も非対称になる（BL-6）。提案の妥当性を評価する際は、変更の表現ではなく、既存状態からの実効的差分の方向を検証すべき。
7. **分析前の中間ラベル生成はアンカリングバイアスを導入する**: 分析ステップの前に変更の性質を記述・分類させると、AI は自己生成したラベルに後続の推論をアンカリングさせる。「自由記述」であっても、テンプレートのヒントや例示が暗黙のカテゴリを提供し、ラベルが判定方向へのショートカットとして機能する（BL-7）。認知的足場の追加は、それが判定方向と相関するラベルを生成させる場合、実質的に判定の非対称操作（原則 #1）と同じ効果を持つ。中間ステップの追加が有効なのは、ラベル付けではなく、証拠の探索や検証を増やす方向の場合に限る。
8. **受動的な記録フィールドの追加は能動的な検証を誘発しない**: テーブルに「関係性を記述する列」を追加しても、AI はもっともらしいテキストを生成するだけで、実際の検証行動（コールパスのトレース、到達性の確認等）を追加的に行うわけではない。記録と検証は異なる認知操作であり、「何を書くか」を変えても「何を調べるか」は変わらない（BL-8）。さらに、記録オーバーヘッドが1関数あたりの探索コストを増加させ、探索予算を圧迫する（原則 #3 の間接的発動）。推論の質を改善するには、記録テンプレートの拡張ではなく、検証行動を直接的に誘発する仕組み（例: 新しい探索ステップの追加）が必要。
9. **メタ認知的自己チェックは自己評価精度の限界により機能しない**: 「自分はトレースしたか？」等の自己チェックを分析プロセスに挿入しても、AI は自分の推論行為の質を正確に自己判定できない（コードを読む行為と具体的入力のトレースの区別が内部的に曖昧であり、常に「やった」に倒れる）。さらに、自己チェックは「本当にやったか？」という疑念を注入するが、この疑念は NOT_EQ の結論（積極的な差異の提示を要する）に対してのみ遅延効果を持ち、EQUIV の結論（消極的な「差異なし」判断）には具体的な追加行動に繋がらない。結果として名目上対称なチェックが実効的には NOT_EQ の立証責任引き上げ（原則 #1, #6）として作用する（BL-9）。自己チェックで推論の質を改善するには、AI の自己評価に依存せず、外部的に検証可能な行動（例: 特定のファイルを読む、特定のテスト入力を明示する）を直接的に要求する必要がある。
10. **必要条件ゲートは失敗モードを弁別しない限り判別力を持たない**: 分析プロセスに条件分岐ゲート（YES/NO → 分岐）を挿入しても、ゲートの条件が判別したい失敗モードと直交していればゲートは形式的に通過されるだけである（BL-10）。例えば「テストが変更コードに到達するか」は relevant test に対してほぼ常に YES であり、「到達しているが結果が変わらない」（EQUIV 偽陽性の本質）を弁別できない。BL-8 の「受動的記録 → 能動的検証へ」という教訓に従い能動的ゲートを設計しても、ゲート条件自体が失敗の原因と異なる次元を検査していれば無効である。検証ステップを挿入する際は「この条件が YES/NO で分かれるとき、判定結果は実際に変わるか？」を事前に検証すべき。さらに、通過率の高いゲートは記述オーバーヘッドのみを残し、探索予算を圧迫する（原則 #3 の間接的発動）。
11. **探索順序の固定は、証拠収集ではなく探索の偏りを生む**: 「まずテストを読め」「まず変更コードを読め」のように探索の開始順序をテンプレートで固定しても、必要な証拠が新たに増えるわけではない。むしろ最初に読む側に注意がアンカリングされ、その側の要約や記録（entry point, setup など）が目的化しやすい（BL-12）。このタスクでは変更差分とテストを往復しながら因果を突き合わせる柔軟な探索が必要であり、片側起点を強制すると反対側の重要信号が後景化する。順序制約は「何を書くか」の規定（原則 #5）や記録フィールド追加（原則 #8）と結びつくと、比較推論を強化しないままオーバーヘッドだけを増やす。改善すべきは開始順序ではなく、往復の中で何を検証するかである。
