# SKILL.md 自動改善パイプライン — 設計資料

## 概要

本パイプラインは、コード推論スキル（SKILL.md）のベンチマーク精度を自動的に改善するシステムである。
2つの AI エージェント（実装者・監査役）が協調し、改善→監査→ベンチマーク→評価→学習のループを回す。

## ゴール

ベンチマーク（20ケース）を5回実行し、2回以上で 100%（20/20）を達成すること。
詳細は Objective.md を参照。

## エージェント構成

| 役割 | ツール | モデル | 責務 |
|------|--------|--------|------|
| 実装者 | Copilot CLI | claude-sonnet-4.6 | 分析、提案、SKILL.md の実装、rationale.md 作成 |
| 監査役 | Claude Code | Claude（ホストのサブスク） | ディスカッション、Audit Rubric 監査、BL 更新 |

### なぜ2つのエージェントを分けるか

- 実装者と監査役を分離することで、自己評価のバイアスを排除する
- 監査役は実装に関与しないため、客観的な評価ができる
- 異なるモデル/ツールを使うことで、思考パターンの多様性を確保する

## 学習メカニズム（★ ステートフル要素）

### failed-approaches.md の自動更新

スコアが前回より下がった場合、監査役（Claude）が以下を自動で行う:

1. 失敗した改善案（proposal.md, diff.patch）を分析
2. 失敗の本質（Fail Core）を Chain-of-Thought で導出
3. failed-approaches.md に新しい BL エントリを追記
4. 既存の共通原則に新たな原則を追加すべきか検討

これにより、パイプラインはイテレーションを重ねるごとに「やってはいけないこと」の知識を蓄積する。

### 共通原則（現時点で導出済み）

1. **判定の非対称操作は必ず失敗する**: 閾値移動はゼロサム
2. **出力側の制約は効果がない**: 改善すべきは入力側と処理側
3. **探索量の削減は常に有害**: 情報不足が誤判定の主因
4. **同じ方向の変更は表現を変えても同じ結果**: 効果の方向が同じなら結果も同じ

## 品質保証メカニズム

### Audit Rubric（7項目、各1-3点）

| # | 項目 | 検査内容 |
|---|------|---------|
| R1 | 汎化性 | 特定ケース・言語に依存しないか |
| R2 | 研究コアの踏襲 | 原論文の構造を維持しているか |
| R3 | 推論プロセスの改善 | 結論ではなくプロセスを改善しているか |
| R4 | 反証可能性の維持 | 反証ステップを弱めていないか |
| R5 | 複雑性の抑制 | 不当に複雑にしていないか |
| R6 | 回帰リスク | 正答ケースを壊さないか |
| R7 | ケース非依存性 | 特定ケースを狙い撃ちしていないか |

合格基準: 全項目 2 以上、かつ合計 14/21 以上

### ディスカッションでのブラックリストチェック

監査役は以下の観点で提案を検証する:
- 表現や用語が違っていても、実質的な効果が同じではないか？
- NOT_EQ と判定するハードルを上げることにならないか？
- 過去の失敗と同じメカニズムに陥っていないか？

### ロールバック機構

- **単発ロールバック**: スコアが前回より下がったら SKILL.md を前の状態に戻す
- **フルロールバック**: 直近3回の平均が 75% 以下なら main の SKILL.md（初期状態）に戻す

## 実行方法

```bash
# Fedora VM 内で実行
cd ~/agentic-code-reasoning-skills

# 1イテレーションだけ実行
bash auto-improve.sh -n 1

# ゴール達成まで実行（最大20イテレーション）
bash auto-improve.sh

# iter-8 から開始して3回実行
bash auto-improve.sh -s 8 -n 3

# バックグラウンドで実行
nohup bash auto-improve.sh > /tmp/auto-improve.log 2>&1 &
```

## 実行環境

- ホスト: Mac mini M4（32GB RAM）
- VM: Fedora 43（Lume, 4CPU / 8GB RAM）
- 実装者: Copilot CLI（`copilot -p --yolo --model claude-sonnet-4.6 -s`）
- 監査役: Claude Code（`claude --dangerously-skip-permissions -p`）
- ベンチマーク: SWE-bench Verified, django/django, 20ペア, Haiku モデル

## 既知の課題

1. **Web 検索が実行されない**: ディスカッションで既存研究の調査を指示しているが、Claude Code の `-p` モードで Web 検索が動作していない
2. **改善方向の収束**: エージェントが NOT_EQ のハードルを上げる方向に収束しやすい。Fail Core と共通原則で抑制を試みている
3. **ベンチマークの確率的変動**: 同じ SKILL.md でもベンチマーク結果が実行ごとに変動する可能性がある（未検証）
